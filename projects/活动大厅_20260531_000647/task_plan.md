## 1. 任务分解

### 1.1 系统策划 (System Planner)

- **任务 1.1.1：活动大厅框架核心逻辑设计**
    - **输入文件**：`活动大厅 - 宏观设计草案.md`
    - **产出文件**：`01_活动大厅_核心框架设计.md`
    - **具体内容**：
        1.  详细定义活动页签状态机（未激活、可参与、已结束）的流转逻辑与判定条件。
        2.  设计活动配置表（`ActivityConfig`）的完整字段定义，包括：`activity_id`, `activity_name`, `start_time`, `end_time`, `unlock_level`, `unlock_chapter`, `sort_weight`, `has_reddot`, `activity_type` (如: `daily_signin`, `limited_instance`, `exchange_shop`), `entry_url` (跳转路径) 等。
        3.  设计活动大厅与各活动子系统的解耦接口规范（如：`IActivityModule` 接口，包含 `onEnter()`, `onExit()`, `getReddotStatus()`, `getUnlockCondition()` 等方法）。
        4.  定义红点上报与传递的标准化流程（主界面 → 大厅入口 → 页签 → 活动内部）。
        5.  明确活动配置表的刷新时机（打开大厅、服务器推送通知）。

- **任务 1.1.2：日签到活动玩法设计**
    - **输入文件**：`活动大厅 - 宏观设计草案.md`
    - **产出文件**：`02_日签到活动_玩法设计.md`
    - **具体内容**：
        1.  设计签到界面的UI布局（奖励列表、角色立绘区域、签到按钮）。
        2.  定义签到数据存储结构（`player_signin_data`），包含：`current_month`, `signed_days_bitmask`, `claimed_reward_tiers`, `missed_days_count`。
        3.  设计签到奖励表（`SigninRewardConfig`），包含：`day_index`, `reward_type`, `reward_id`, `reward_amount`。
        4.  设计连续签到奖励逻辑（7/14/21/全月）。
        5.  设计补签与加速签到的逻辑，明确调用接口与数据变更。

### 1.2 数值策划 (Numerical Planner)

- **任务 1.2.1：日签到奖励数值设计**
    - **输入文件**：`02_日签到活动_玩法设计.md`
    - **产出文件**：`03_日签到_奖励数值表.xlsx`
    - **具体内容**：
        1.  设计每日签到奖励的具体数值（金币、经验材料、单抽券碎片数量）。
        2.  设计连续签到奖励（限定头像框、角色碎片）的价值与产出节奏。
        3.  计算每月通过签到产出的抽卡资源总量，确保在5-10抽之间。
        4.  设计补签价格递增曲线（如：第1天10钻，第2天20钻，第N天 `N*10` 钻，并设置上限）。
        5.  设计签到加速的固定价格（如100钻）。

### 1.3 格式翻译 (Schema Translator)

- **任务 1.3.1：活动配置表 JSON Schema 生成**
    - **输入文件**：`01_活动大厅_核心框架设计.md`
    - **产出文件**：`04_活动配置表_Schema.json`
    - **具体内容**：将系统策划定义的活动配置表字段，翻译为严格的 JSON Schema 格式，用于校验运营后台下发的配置数据。

- **任务 1.3.2：日签到数据存储 JSON Schema 生成**
    - **输入文件**：`02_日签到活动_玩法设计.md`
    - **产出文件**：`05_日签到数据_Schema.json`
    - **具体内容**：将玩家签到数据存储结构翻译为 JSON Schema，用于客户端本地存储与服务器数据校验。

### 1.4 程序执行 (Code Agent)

## 2. 执行顺序与依赖

```mermaid
graph TD
    subgraph 第一阶段 (并行设计)
        A[系统策划: 活动大厅框架] -->|产出: 01| B[系统策划: 日签到活动]
        A -->|产出: 01| C[UI Agent: 活动大厅UI]
        B -->|产出: 02| D[UI Agent: 日签到UI]
        B -->|产出: 02| E[数值策划: 奖励数值]
    end

    subgraph 第二阶段 (格式转换)
        A -->|产出: 01| F[格式翻译: 活动配置表Schema]
        B -->|产出: 02| G[格式翻译: 日签到数据Schema]
    end

    subgraph 第三阶段 (并行开发)
        F -->|产出: 04| H[程序执行: 活动大厅框架]
        G -->|产出: 05| I[程序执行: 日签到活动]
        E -->|产出: 03| I
        C -->|产出: 06| H
        D -->|产出: 07| I
    end

    subgraph 第四阶段 (审查与集成)
        E -->|产出: 03| J[审查官: 数值审查]
        A -->|产出: 01| K[审查官: 系统设计审查]
        B -->|产出: 02| K
        H -->|产出: 代码| L[集成测试]
        I -->|产出: 代码| L
    end
```

**依赖关系总结：**
- **串行**：系统策划 → 格式翻译 → 程序执行。系统策划必须先完成设计，格式翻译才能生成 Schema，程序才能开始编码。
- **并行**：系统策划的两个子任务（框架与日签到）可以并行。UI设计可以与系统策划并行。数值策划可以与系统策划并行。审查官可以与程序开发并行（审查设计文档）。
- **阻塞点**：程序执行必须等待格式翻译和UI设计稿完成。集成测试必须等待所有程序模块完成。

## 3. 风险提示

1.  **活动配置表格式对齐风险**：
    - **阻塞点**：任务 1.3.1（格式翻译）产出 JSON Schema 后，需要与运营团队确认。若运营后台无法支持该 Schema 格式，或字段定义有歧义，将导致整个活动大厅框架无法上线。
    - **建议**：在系统策划设计阶段（任务 1.1.1），就应邀请运营团队参与评审，确保配置表字段定义清晰、易于理解，并预留扩展字段。

2.  **日签到补签定价风险**：
    - **阻塞点**：任务 1.2.1（数值策划）设计的补签价格曲线，若未经审查官（任务 1.7.1）严格审查，可能导致玩家付费意愿极低（价格过高）或严重冲击付费抽卡收入（价格过低）。
    - **建议**：数值策划在设计时需参考同类游戏数据，并与审查官进行多轮沟通。审查报告应明确指出价格曲线的合理区间。

3.  **活动大厅性能风险**：
    - **阻塞点**：任务 1.4.1（程序执行）在实现页签栏滑动与背景插画切换时，若未做性能优化，在低端设备上可能出现卡顿。
    - **建议**：程序实现时需考虑资源预加载、对象池、LOD（细节层次）等技术。UI设计（任务 1.5.1）应限制背景插画的分辨率与复杂度。

4.  **跨团队依赖冲突**：
    - **阻塞点**：活动大厅需要与**邮件系统**（任务 5.3）、**红点系统**（任务 5.4）进行数据联动。若这些系统的接口尚未稳定或变更，将导致活动大厅开发受阻。
    - **建议**：在项目启动初期，PM 应组织相关系统负责人进行接口对齐会议，明确接口定义、调用方式与异常处理。将接口联调作为关键里程碑进行跟踪。
