# 背包系统 - 宏观设计草案

## 一、系统概述与设计愿景

### 1.1 系统定位
- **业务边界**：纯功能型收纳箱，不承担任何角色展示、互动或情感反馈职责。
- **前置条件**：玩家完成新手引导后解锁背包入口。
- **流转逻辑**：玩家从主界面或任意需要道具操作的子界面（如商店、强化、送礼）调用背包，以列表/网格形式展示已拥有的道具。
- **数据状态变化**：背包数据为全局共享容器，任何系统的道具增减均通过背包接口完成。
- **边界与异常兜底**：背包容量满时，新获得道具自动进入邮件系统（溢出邮件），背包界面显示容量提示。

### 1.2 设计目标
- **前置条件**：系统需与道具定义系统、邮件系统、商店系统、强化系统、好感度系统、战斗结算系统完成数据对接。
- **流转逻辑**：所有道具的获取、消耗、出售、使用均以背包为唯一数据源，其他系统通过接口读写背包数据。
- **数据状态变化**：每次操作后，背包内对应道具的数量、标签状态、过期时间等字段实时更新。
- **边界与异常兜底**：若操作失败（如网络中断、数据校验不通过），背包数据回滚至操作前状态，并返回错误码。

## 二、核心规则与玩法机制

### 2.1 道具分类与标签系统
- **分类维度**：按道具类型分为「养成材料」「活动道具」「任务道具」「礼物」「皮肤」「消耗品」等大类，每个大类下可细分二级分类。
- **标签系统**：每个道具实例可携带多个类型标签（如 `MATERIAL_ASCENSION`、`EVENT_EXCHANGE`、`GIFT_PREFERRED`），标签作为后续系统（如出售筛选、合成配方、活动兑换）的判断条件。
- **流转逻辑**：道具定义时由策划配置基础标签，玩家获得道具时系统根据当前活动/角色状态动态追加临时标签（如 `EVENT_BOOSTED`）。
- **数据状态变化**：标签字段为数组，支持增删改查，标签变更不影响道具ID或数量。

### 2.2 道具使用功能
- **前置条件**：道具必须配置 `usable` 属性为 `true`，且玩家满足使用条件（如等级、任务阶段）。
- **流转逻辑**：玩家选中道具 → 点击“使用” → 系统校验条件 → 执行使用效果（如增加体力、打开宝箱、兑换皮肤） → 扣除道具数量。
- **数据状态变化**：使用成功后，道具数量减少，对应效果数据（如体力值、皮肤解锁状态）更新。
- **边界与异常兜底**：若使用条件不满足，弹出提示并阻止操作。若使用后道具数量为0，自动从背包中移除该道具条目。

### 2.3 道具出售功能
- **前置条件**：道具必须配置 `sellable` 属性为 `true`，且未锁定（如任务道具、已装备皮肤不可出售）。
- **流转逻辑**：玩家进入出售界面 → 按标签/分类筛选可出售道具 → 选择数量 → 确认出售 → 获得对应货币（基础货币/活动代币）。
- **数据状态变化**：出售后，道具数量减少，玩家货币余额增加。
- **边界与异常兜底**：若道具为不可出售类型，出售界面中该道具置灰或隐藏。出售操作不可逆，需二次确认弹窗。

### 2.4 批量操作
- **批量出售**：支持按标签全选（如“一键出售所有N品质材料”），或手动多选后批量出售。
- **批量使用**：支持输入数量或滑动条，一次性使用多个相同道具（如一次性开启10个宝箱）。
- **数据状态变化**：批量操作视为多次单次操作的原子化合并，若中间某次失败，已成功部分不回溯。

## 三、表现层与角色展示联动

### 3.1 纯功能型UI
- **UI设计原则**：简洁高效，以列表/网格展示道具图标、名称、数量、标签。不包含角色模型、看板娘、互动动画。
- **操作反馈**：点击、滑动、筛选等操作仅提供标准UI反馈（高亮、弹窗、飘字），不插入任何角色特写或互动演出。
- **边界与异常兜底**：若道具图标缺失，显示默认占位图。若道具数量为0，该道具条目自动隐藏。

### 3.2 角色展示零介入
- **强制规则**：背包系统在任何界面、任何操作中均不调用角色模型、不播放角色动画、不触发角色语音。
- **例外处理**：若其他系统（如送礼、换装）需要从背包选择道具，角色展示由调用方系统负责，背包仅提供数据容器。

## 四、经济循环与商业化埋点

### 4.1 免费产出
- **前置条件**：玩家通过战斗、任务、活动等途径获得道具，存入背包。
- **流转逻辑**：背包不直接产出任何资源，仅作为存储与中转容器。
- **数据状态变化**：道具数量增加，背包容量占用增加。
- **边界与异常兜底**：若背包容量不足，道具进入溢出邮件，邮件保留30天后自动删除。

### 4.2 付费埋点
- **前置条件**：玩家在商城中购买道具礼包、扩容卡等。
- **流转逻辑**：购买后，道具直接进入背包，扩容卡使用后永久增加背包容量上限。
- **数据状态变化**：背包道具数量增加，`inventory_capacity_max` 增加对应数值。
- **边界与异常兜底**：若购买时背包已满，道具进入邮件。扩容卡使用后容量立即生效。

### 4.3 资源循环
- **前置条件**：背包系统不产出抽卡资源、不消耗付费货币。
- **流转逻辑**：所有道具的获取与消耗均通过外部系统（战斗、任务、商店、活动）完成，背包仅记录状态变化。
- **数据状态变化**：无抽卡资源相关数据变化。
- **边界与异常兜底**：若玩家尝试通过背包系统直接获取抽卡资源，系统提示“该功能不产出抽卡资源”。

## 五、旧系统与数据联动

### 5.1 道具定义系统
- **前置条件**：所有道具必须先在 `item_definition` 表中注册，包含ID、名称、图标、品质、类型、标签、是否可出售、是否可使用、使用效果ID等字段。
- **流转逻辑**：背包系统读取 `item_definition` 表渲染道具信息，操作时校验 `sellable`、`usable` 等属性。
- **数据状态变化**：道具定义变更（如新增标签、修改品质）后，背包中已有道具的标签字段需同步更新（通过定时任务或事件驱动）。
- **边界与异常兜底**：若道具定义被删除，背包中对应道具显示为“未知道具”，无法进行任何操作。

### 5.2 邮件系统
- **前置条件**：背包容量满时，新获得道具自动触发溢出邮件。
- **流转逻辑**：系统创建一封包含溢出道具的邮件，邮件标题为“背包已满，道具暂存”，邮件保留30天。
- **数据状态变化**：邮件系统新增一条记录，背包数据不变。
- **边界与异常兜底**：若邮件系统也满，道具丢失并记录错误日志。玩家从邮件领取道具时，若背包仍满，再次触发溢出邮件。

### 5.3 商店系统
- **前置条件**：玩家在商店购买道具或出售道具。
- **流转逻辑**：购买时，商店调用背包接口增加道具；出售时，背包调用货币系统接口增加货币。
- **数据状态变化**：背包道具数量变化，货币余额变化。
- **边界与异常兜底**：若购买时背包满，提示“背包已满，请先清理”。若出售时网络中断，货币不增加，道具不扣除。

### 5.4 强化/好感度/战斗结算系统
- **前置条件**：这些系统在消耗或产出道具时，均通过背包接口完成。
- **流转逻辑**：消耗时，背包校验数量并扣除；产出时，背包校验容量并增加。
- **数据状态变化**：背包道具数量实时更新。
- **边界与异常兜底**：若消耗时道具不足，返回错误码，调用方系统显示“道具不足”提示。

## 六、待确认风险与追问

### 6.1 标签系统复杂度
- **前置条件**：标签需支持动态追加（如活动期间临时标签）。
- **流转逻辑**：标签由策划在 `item_definition` 中配置基础标签，活动系统通过事件驱动追加临时标签，活动结束后自动移除。
- **数据状态变化**：`item_tags` 字段为数组，支持增删改查。
- **边界与异常兜底**：若活动配置错误导致标签未及时移除，需手动执行清理脚本。

### 6.2 批量操作性能
- **前置条件**：玩家一次性出售/使用大量道具（如1000个）。
- **流转逻辑**：批量操作拆分为多次单次操作，但需保证原子性（全部成功或全部失败）。
- **数据状态变化**：若中间某次失败，已成功部分不回溯，需记录失败日志。
- **边界与异常兜底**：单次批量操作上限为999个，超过则分多次执行。

### 6.3 道具过期处理
- **前置条件**：部分活动道具设有过期时间。
- **流转逻辑**：背包系统每日首次打开时，检查所有道具的 `expire_time` 字段，过期道具自动删除并发送邮件通知。
- **数据状态变化**：过期道具数量减少，邮件系统新增通知邮件。
- **边界与异常兜底**：若玩家在道具过期前已离线，上线时统一清理。过期道具不可恢复。
