# 角色私人宿舍系统 - WBS 任务拆解计划

## 1. 任务分解

### System Planner (系统策划)
- **任务1.1**: 细化“外宣展示内容”的模块清单
  - 输入: 终审设计草案
  - 产出: `宿舍外宣展示内容清单.md` (仅列出展示类型: 角色展示、互动动画、换装展示、场景预览，不包含达成逻辑)
- **任务1.2**: 定义“操作展示”的交互流程
  - 输入: 设计草案 + 外宣清单
  - 产出: `宿舍操作展示交互规范.md` (描述如何操作展示: 点击触发、拖拽、视角切换等，不含具体动画逻辑)

### Numerical Planner (数值策划)
- **任务2.1**: 设计宿舍买断定价表
  - 输入: 设计草案(定价范围30-60元)
  - 产出: `宿舍定价方案.xlsx` (单角色定价、礼包定价、通行证定价)
- **任务2.2**: 定义好感度任务奖励的家具/装饰品数值
  - 输入: 设计草案(场景自定义)
  - 产出: `家具奖励数值表.xlsx` (任务完成条件、奖励物品ID、解锁条件)

### Schema Translator (格式翻译)
- **任务3.1**: 将外宣展示清单翻译为 JSON Schema
  - 输入: `宿舍外宣展示内容清单.md`
  - 产出: `宿舍展示内容_schema.json`
- **任务3.2**: 将操作展示交互规范翻译为 JSON Schema
  - 输入: `宿舍操作展示交互规范.md`
  - 产出: `宿舍操作展示_schema.json`

### Code Agent (程序执行)
- **任务4.1**: 实现宿舍场景加载与角色展示
  - 输入: `宿舍展示内容_schema.json`
  - 产出: `宿舍_scene.gd` (场景加载、角色渲染、摄像机控制)
- **任务4.2**: 实现操作展示交互逻辑
  - 输入: `宿舍操作展示_schema.json`
  - 产出: `宿舍_interaction.gd` (点击触发、拖拽、视角切换)
- **任务4.3**: 实现换装展示功能
  - 输入: 服装系统数据接口
  - 产出: `宿舍_wardrobe.gd` (服装切换、展示动作触发)

### UI Agent (UX/UI 设计)
- **任务5.1**: 设计外宣展示页面的 UI 布局
  - 输入: `宿舍外宣展示内容清单.md`
  - 产出: `宿舍外宣UI设计稿.png` (极简UI、角色展示区、互动提示图标)
- **任务5.2**: 设计操作展示的交互控件
  - 输入: `宿舍操作展示交互规范.md`
  - 产出: `宿舍操作UI设计稿.png` (视角切换按钮、触摸区域提示、换装菜单)

### Audit Agent (审查官)
- **任务6.1**: 审查外宣展示内容是否符合擦边底线
  - 输入: `宿舍外宣展示内容清单.md` + `宿舍操作展示交互规范.md`
  - 产出: `宿舍外宣审查报告.md` (确认无直接性行为/露点内容)
- **任务6.2**: 审查定价方案的经济平衡性
  - 输入: `宿舍定价方案.xlsx`
  - 产出: `宿舍定价审查报告.md` (确认不影响核心经济循环)

### Combat Agent (战斗策划)
- **任务7.1**: 定义宿舍互动动画的触发参数
  - 输入: 设计草案(互动触发部分)
  - 产出: `宿舍互动动画参数.json` (触摸部位、反馈动画ID、语音ID、表情ID)

## 2. 执行顺序与依赖

### 串行依赖链
```
[System Planner] 任务1.1 → [Schema Translator] 任务3.1 → [Code Agent] 任务4.1
[System Planner] 任务1.2 → [Schema Translator] 任务3.2 → [Code Agent] 任务4.2
[Numerical Planner] 任务2.1 → [Audit Agent] 任务6.2
[Combat Agent] 任务7.1 → [Code Agent] 任务4.3
```

### 可并行任务
- **并行组A**: 任务1.1 + 任务1.2 (System Planner 可同时产出两份文档)
- **并行组B**: 任务2.1 + 任务2.2 (Numerical Planner 可同时处理定价和奖励)
- **并行组C**: 任务3.1 + 任务3.2 (Schema Translator 可同时翻译两份 Schema)
- **并行组D**: 任务4.1 + 任务4.2 + 任务4.3 (Code Agent 可并行实现三个模块)
- **并行组E**: 任务5.1 + 任务5.2 (UI Agent 可同时设计两份 UI)
- **并行组F**: 任务6.1 + 任务6.2 (Audit Agent 可同时审查两份内容)

### 整体执行流程图
```
Week1: [System Planner] 任务1.1, 1.2 | [Numerical Planner] 任务2.1, 2.2 | [Combat Agent] 任务7.1
Week2: [Schema Translator] 任务3.1, 3.2 | [Audit Agent] 任务6.1, 6.2 | [UI Agent] 任务5.1, 5.2
Week3: [Code Agent] 任务4.1, 4.2, 4.3
Week4: 集成测试与修复
```

## 3. 风险提示

### 阻塞点
1. **System Planner 产出延迟**: 任务1.1和1.2是后续所有任务的输入，若产出延迟将导致整个项目延期。建议设置硬性截止日期(如2天)。
2. **Schema Translator 格式冲突**: 任务3.1和3.2的 Schema 可能因字段命名不一致导致 Code Agent 实现困难。建议提前定义统一命名规范。

### 跨团队依赖冲突
1. **Code Agent 与 UI Agent 的交互冲突**: 任务4.2(交互逻辑)和任务5.2(交互控件)需紧密配合，若 UI 设计稿与代码实现不匹配，需反复沟通。建议每周同步一次。
2. **Combat Agent 与 Code Agent 的参数对齐**: 任务7.1输出的动画参数 JSON 需与任务4.3的换装展示逻辑完全一致，否则动画无法触发。建议 Combat Agent 提前与 Code Agent 确认参数格式。
3. **Audit Agent 审查结果导致返工**: 若任务6.1发现擦边内容超标，System Planner 需修改任务1.1和1.2，导致后续所有任务重做。建议在 System Planner 产出前先进行预审查。

### 技术风险
- **移动端性能**: 任务4.1的高精度渲染可能超出低配机型能力，需预留性能优化时间(如 LOD 模型、纹理压缩)。
- **内容生产量**: 每个角色需独立制作互动动画，若首发角色过多，Combat Agent 的任务7.1可能成为瓶颈。建议首发角色控制在3-5个。
