# 公会科技树系统 - 数据验收表

## 一、顶层字段

| 字段名 | 含义 | 数据类型 | 必填 | 取值范围/约束 | 防漏洞措施 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| `guild_id` | 公会唯一ID | 字符串 | 是 | 1-64个字符 | 防止不同公会数据混淆 |
| `tech_tree_version` | 科技树版本号 | 整数 | 是 | >= 1 | 版本更新时，旧数据可据此进行迁移，防止数据结构不兼容 |
| `nodes` | 所有科技树节点状态 | 对象 | 是 | 至少包含1个节点 | 防止空科技树 |
| `currency_log` | 货币消耗日志 | 数组 | 是 | 元素唯一 | 用于审计和回滚，防止重复消耗 |
| `trigger_trace` | 全局触发来源 | 对象 | 是 | 见下方 | **核心防漏洞**：记录本次数据变更的最终来源，用于追踪异常操作 |

## 二、节点状态 (`nodes` 下的每个节点)

| 字段名 | 含义 | 数据类型 | 必填 | 取值范围/约束 | 防漏洞措施 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| `node_id` | 节点ID | 字符串 | 是 | 格式：`tech_node_`开头，后跟字母、数字、下划线 | 防止非法节点ID |
| `current_level` | 当前等级 | 整数 | 是 | 0-10 (0=未解锁) | 防止等级溢出，限制最大等级 |
| `is_unlocked` | 是否解锁 | 布尔 | 是 | true/false | 与`current_level`联动，防止状态不一致 |
| `unlock_timestamp` | 解锁时间戳 | 整数 | 是 | >= 0 (Unix秒) | 用于时间线审计 |
| `last_upgrade_timestamp` | 最后升级时间戳 | 整数 | 是 | >= 0 (Unix秒) | 用于时间线审计 |

## 三、货币消耗日志 (`currency_log` 数组中的每个元素)

| 字段名 | 含义 | 数据类型 | 必填 | 取值范围/约束 | 防漏洞措施 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| `log_id` | 日志唯一ID | 字符串 | 是 | 格式：`currency_log_` + 32位十六进制数 | **防重复提交**：保证每条日志唯一 |
| `node_id` | 被操作节点 | 字符串 | 是 | 同节点ID格式 | 确保操作对象合法 |
| `action_type` | 操作类型 | 枚举 | 是 | `"unlock"` 或 `"upgrade"` | 防止非法操作 |
| `cost_amount` | 消耗数量 | 整数 | 是 | 1-1000 | 防止单次消耗过大或为0 |
| `cost_timestamp` | 消耗时间戳 | 整数 | 是 | >= 0 (Unix秒) | 用于时间线审计 |
| `trigger_source` | 触发来源标签 | 字符串 | 是 | 格式：`guild_<公会ID>_member_<成员ID>_<操作>` | **核心防漏洞**：精确追踪到是哪个公会的哪个成员触发了消耗，防止地图边缘爆炸等异常来源导致无限刷货币 |

## 四、全局触发来源 (`trigger_trace`)

| 字段名 | 含义 | 数据类型 | 必填 | 取值范围/约束 | 防漏洞措施 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| `source_type` | 来源类型 | 枚举 | 是 | `"guild_member"`, `"system"`, `"admin_tool"` | 区分正常玩家、系统自动、管理员操作 |
| `source_id` | 来源ID | 字符串 | 是 | 至少1个字符 | 具体到人/系统模块 |
| `action` | 具体动作 | 枚举 | 是 | `"unlock_node"`, `"upgrade_node"`, `"system_reset"`, `"admin_modify"` | 明确做了什么 |
| `timestamp` | 触发时间戳 | 整数 | 是 | >= 0 (Unix秒) | 记录操作发生时间 |

## 五、设计上的防漏洞措施总结

1.  **因果追踪**：`currency_log` 中的 `trigger_source` 字段，强制要求记录“谁”在“哪个公会”做了“什么操作”。任何没有合法来源标签的货币消耗请求都会被拒绝，从根本上杜绝了“地图边缘爆炸导致无限刷怪”这类无来源的异常事件。
2.  **防重复提交**：`currency_log` 的 `log_id` 使用唯一ID，确保同一次消耗不会被重复处理。
3.  **数据一致性**：`nodes` 中的 `is_unlocked` 和 `current_level` 联动，防止出现“已解锁但等级为0”或“未解锁但等级>0”的矛盾状态。
4.  **数值安全**：`cost_amount` 限制在1-1000，`current_level` 限制在0-10，防止数值溢出或异常。
5.  **版本兼容**：`tech_tree_version` 字段确保在游戏更新科技树结构时，可以安全地迁移旧数据，避免数据损坏。
6.  **全局审计**：`trigger_trace` 记录了本次数据变更的最终来源，配合 `currency_log`，可以形成完整的操作审计链。
